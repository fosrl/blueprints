# __RESOURCE_NAME__

This is the starting point for the `__SLUG__` blueprint.

## Required Variables

- `BLUEPRINT_NAME`: stable blueprint slug used in labels.
- `RESOURCE_NAME`: human-readable Pangolin resource name.
- `SERVICE_SUBDOMAIN`: subdomain prefix only.
- `SERVICE_CONTAINER_NAME`: the public Compose service key and container name Pangolin should target.
- `SERVICE_PORT`: internal port Pangolin should reach.

## Env Placeholder Rules

- Use `CHANGE_ME` for values the user must edit manually.
- Use `GENERATE_<IDENTIFIER>` for secrets that `./bin/blueprint init <service>` should generate.
- Reuse the same `GENERATE_<IDENTIFIER>` token when multiple keys must share the same generated value.
- Use `RESOURCE_AUTH_*` for blueprint-specific auth overrides and additions.
- Use the root `.env` for shared `GLOBAL_AUTH_*` defaults.

## Hostname Pattern

The public hostname is derived from the shared root `.env`:

```text
${SERVICE_SUBDOMAIN}.${BASE_DOMAIN}
```

That keeps contributor blueprints portable across different Pangolin deployments.

The template uses Pangolin’s Docker label format directly:

```text
pangolin.public-resources.<blueprint-id>.*
```

For HTTP resources, keep `name`, `protocol`, `full-domain`, and `targets[0].method` together or Pangolin will reject the blueprint.

Auth labels such as `auth.sso-roles[0]` and `auth.whitelist-users[0]` are generated by `./bin/blueprint` from `GLOBAL_AUTH_*` and `RESOURCE_AUTH_*` env variables, rather than being hard-coded into the Compose file.

## Suggested Workflow

Create a new blueprint with:

```bash
./bin/blueprint new __SLUG__
```

Then test it with:

```bash
./bin/blueprint init __SLUG__
./bin/blueprint auth __SLUG__
./bin/blueprint config __SLUG__
```
